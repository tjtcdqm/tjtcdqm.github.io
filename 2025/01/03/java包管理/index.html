java 包管理机制

从C/C++到Java总是有几个地方不习惯，其中Java的包管理这种在c/c++中完全不存在的概念更是c/c++人感到疑惑。为什么编译的时候每个类都会编译成class文件？我要怎么使用另一个文件的类，为什么import了还不行，难道“链接”有问题？package关键字又是什么，为什么我在一个文件中使用了import 某个package，还是不能使用另一个文件？为什么执行了的时候只需要指定主类？为什么我换一个路径执行这个主类又出错了？总之，种种问题，都是因为不了解Java的包管理机制和运行机制导致的。

答案是不能。为什么？我不是指定了正确的路径和主类名字吗？为什么不能运行？

其实java这个命令行工具的参数是一个主类的名字，而不是一个路径的名字。
也就是说，你在执行java ./java/HelloWorld的时候java会认为这个主类叫“./java/HelloWorld”，但是他并没有找到这个主类。

那么Java会去哪里找这个类呢？去环境变量classpath中的路径查找。如果我在classpath种添加一个./java路径，那么我就可以正确执行这个程序。但是命令仍然不是java ./java/HelloWorld而是java HelloWorld

java会按这个主类的名字去classpath中的路径查找这个主类，如果找到就会加载他并执行。又因为java要求文件名和public类的类名相同，所以查找还是有一定逻辑的。

这又是为什么，我不是正确地引用了包名，并正确定义了包吗？其实正确定义包除了使用package关键字外，还要把它放到相应的文件夹中。

使用package com之后，要把这个文件放到一个叫com的目录下，在哪里编译处字节码文件。然后在主目录中编译主类，并把主类的名字告诉java才行。

还是哪个问题，为什么我只要告诉java主类就行了？我不是使用了其他的包吗？为什么java程序不关心这个外部类的位置也能正确执行程序。

说是不关心，还是要关心的。怎么知道这个外部类的磁盘位置呢？答案是包名就包含了外部类的位置。使用javac编译的时候，javac会将标识符SayHello变为全限定名com.SayHello，然后编译后的字节码文件中是不会包含import语句的，即import语句只是编译时使用，运行时不需要。

在运行时，遇到com.SayHello这个全限定名，就会去com/目录下找这个类相应的字节码。也就是说，包名不仅指定了一个类属于哪个包，还指定了类的字节码文件所在的位置。所以对应的包应该放在对应的目录中，jvm才能正确查找字节码文件，才能正确加载类。那么这个com/目录又在哪个目录下找呢？很简单，在classpath的路径中找就可以了。

jar包

为了像c/c++一样，实现头文件和库，java采用了jar包。这是一种基于zip的压缩格式，能将很多的包压缩到一个压缩包中，供给第三方使用。

而且jar包还可以变为可执行的jar包。这就需要手动设置jar包的主类。同时也有类似静态链接的fat jar包，即所有的依赖都在jar包中了，不需要而外指定依赖。也有类似动态依赖的jar包，这就需要手动为他添加-cp，来为它的依赖提供一个本机的查找路径。
